# Сценарии нагрузки Firestore (ручная проверка)

Цель: оценить reads/writes до масштаба «тысячи пользователей» и поймать регрессии в синке.

## Базовые метрики

- Firebase Console → **Firestore** → **Usage** (reads/writes за интервал).
- Локально воспроизведите сценарий и сузьте окно времени до минут.

## Сценарии

1. **Холодный старт**  
   Очистка site data → вход → открытие дерева. Зафиксировать reads до первого idle.

2. **Много вкладок**  
   Две вкладки одного пользователя: правки в одной, фокус на другой. Убедиться, что нет лишних полных `getDocs` (см. [ARCHITECTURE.md](ARCHITECTURE.md)).

3. **Большое дерево**  
   Импорт или генерация N узлов (например 500–2000), затем первый синк и повторный фокус окна. Сверить с ожиданием: один полный снимок при необходимости, без двойного `getDocs` в `syncAllNodesToFirestore`, если передан `cloudNodes`.

4. **Сбой облака и восстановление**  
   Симуляция `permission-denied` / офлайн → снова онлайн; проверить flush pending push и счётчик reads.

5. **Журнал изменений**  
   Частые мелкие правки: рост документов в `security`, drain + listener; убедиться, что cleanup журнала отрабатывает.

## Целевые SLO (заполните под продукт)

| Метрика | Цель |
|---------|------|
| Открытие среднего роадмапа (p95) | _мс_ |
| Reads на одну «типичную» сессию 30 мин | _число_ |
| Стоимость на DAU | _валюта/мес_ |
